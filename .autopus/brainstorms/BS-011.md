## 추천 방향

**C2 (Hook-triggered File Read)** 를 1순위로 추천합니다.

**근거:**
1. 이미 hook 인프라가 존재 (claude stop hook, gemini afteragent, opencode plugin)
2. 출력 hook의 대칭으로 입력 hook 추가 — 아키텍처 일관성
3. 파일 감시(fsevents/inotify) 불필요 — hook 실행 시점에 파일 읽기
4. paste-buffer 완전 우회 — surface 상태 무관
5. 구현 범위가 가장 작음 (각 hook 스크립트에 input 읽기 로직 추가)

**단기 vs 중장기 전략:**
- 단기 (현재): SPEC-ORCH-016 paste-buffer 패치 (recreatePane + backoff + paneID 참조 수정)
- 중기: C2 Hook-triggered File Read 구현
- 장기: A1 (LSP JSON-RPC) 또는 E5 (단일 sidecar)로 프로토콜 표준화

**"각 CLI를 미리 띄워놓는 방식 (R3)"** 에 대해:
- 가능하지만 UX 부담이 큼 (사용자가 3개 터미널을 미리 준비해야 함)
- Orchestra의 장점인 "한 명령으로 모든 프로바이더 자동 실행"이 퇴화
- C2와 조합하면 유효: 미리 띄운 CLI의 hook이 input 파일을 감시
