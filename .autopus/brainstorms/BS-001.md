### 분석 과정

**현재 아키텍처 한계**: `pkg/orchestra/runner.go`에서 각 프로바이더는 `exec.CommandContext("claude", "--print", prompt)` 형태의 one-shot 실행. 코드 탐색/도구 사용 불가, 프로바이더 간 맥락 공유 없음.

**4가지 접근 탐색**:
1. **Agentic One-Shot Relay** — 도구 접근 권한 포함 one-shot 실행 + 릴레이
2. **Pane Interactive Relay** — cmux pane에서 인터랙티브 세션 + send-keys 릴레이
3. **Hybrid Detach + Relay** — SPEC-ORCH-003 detach + relay chain 결합
4. **Context File Protocol** — 프로바이더 간 구조화된 파일 기반 컨텍스트 공유 프로토콜

### Idea 1: Agentic One-Shot Relay
각 프로바이더를 도구 접근 권한 포함 one-shot으로 실행하되, 릴레이로 이전 결과 전달.
```
claude -p "분석해줘" --allowedTools Read,Grep,Bash → /tmp/relay/claude.md
codex --quiet "$(cat /tmp/relay/claude.md) 참고하여 분석" → /tmp/relay/codex.md
gemini -p "종합 리뷰" ← claude.md + codex.md
```
- 기존 runner.go 패턴 확장으로 구현 가능
- CLI별 agentic 플래그 지원 여부 확인 필요

### Idea 2: Pane Interactive Relay
cmux/tmux pane에서 인터랙티브 세션 시작, send-keys로 릴레이.
- 가장 깊은 맥락 (멀티턴, 도구 자유 사용)
- pane 관리/완료 감지 복잡도 높음

### Idea 3: Hybrid Detach + Relay
SPEC-ORCH-003 detach + relay chain 결합. job.json에 relay 필드 추가.
- 비동기 + 릴레이 결합
- 상태 머신 복잡도

### Idea 4: Context File Protocol
`.autopus/relay/{jobID}/` 디렉토리에 구조화된 분석 결과 파일 규약.
- 실행 방식(one-shot/interactive/detach)과 독립적인 프로토콜 레이어
- 점진적 도입 가능, 구현 단순

## 추천 방향

**2단계 접근 추천**:
- **Phase A**: Context File Protocol 도입 — 프로바이더 간 구조화된 결과 공유 프로토콜 (`relay` strategy 추가)

Phase A만으로도 현재 대비 품질 향상 (이전 프로바이더의 분석을 참조할 수 있음). Phase B에서 프로바이더가 직접 코드를 읽으면 획기적 품질 도약.
