## 원본 아이디어
- What: autopus-adk --multi 플래그가 여러 프로바이더를 pane 인터랙티브 + hook 파일 시그널로 오케스트레이션
- Why: 기존 ReadScreen 화면 스크래핑 방식이 불안정하고 결과 파싱이 불가능. hook 기반으로 전환하면 깨끗한 결과 수집 + slash command 지원 + OAuth 무료 활용이 모두 가능
- Who: 로컬 개발자가 여러 AI 프로바이더를 동시에 활용하여 debate/relay/consensus
- When: 차기 릴리즈 (SPEC-ORCH-006 확장)

## 업계 리서치 요약 (2026-03-26~27)

### 결과 수집 — 아무도 화면 스크래핑 안 함

업계 도구(claude-squad, AMUX, ittybitty, NTM, Agent Teams)는 모두:
- pane은 "사람이 보는" 모니터링 전용
- 결과는 git diff, JSONL 파싱, SQLite, 서브에이전트 반환값 등 별도 채널로 수집

### CLI headless vs SDK vs pane+hooks

| | slash cmd | 멀티턴 | OAuth무료 | 결과수집 |
|---|---|---|---|---|
| CLI -p --resume | **불가** | O | O | O (JSON) |
| SDK | O | O | **불가** (API Key 필수) | O |
| pane + hooks | **O** | **O** | **O** | **O** |

pane + hooks만이 4가지 요구사항을 모두 충족.

## 확정 아키텍처: 통일 패턴

### 핵심 결정

**3개 프로바이더 동일 플로우: pane 인터랙티브 + hook/plugin 파일 시그널**

```
auto orchestra --multi "프롬프트"
  → Claude pane + Gemini pane + opencode pane 생성
  → 각 pane에 send-keys로 프롬프트 주입
  → 사람이 각 pane TUI를 모니터링 가능
  → 각 hook/plugin이 완료 시 결과를 파일에 기록
  → 오케스트레이터가 파일 감지
  → 결과 수집 → debate/relay/consensus 실행
```

### 프로바이더별 hook/plugin 매핑

| Provider | CLI | Hook 이벤트 | 응답 필드 | 설정 위치 |
|----------|-----|------------|-----------|-----------|
| Claude Code | claude | Stop hook | last_assistant_message | .claude/settings.json |
| Gemini CLI | gemini | AfterAgent hook | prompt_response | .gemini/settings.json |
| opencode | opencode | experimental.text.complete plugin | text | opencode.json (plugin) |

### 파일 시그널 프로토콜

```
완료 시 hook/plugin이 기록:
  /tmp/autopus/{session-id}/result.json
    { session_id, provider, response, timestamp }
  /tmp/autopus/{session-id}/done  (시그널 파일)

오케스트레이터가 감지:
  inotify 또는 polling으로 done 파일 감지
  → result.json 파싱 → 깨끗한 응답 텍스트 획득
```

## 오케스트레이션 전략별 동작

### Consensus (기본)
```
3개 pane 병렬 실행 → 각 hook 결과 수집 → 66% 투표 → 최종 결과
```

### Debate
```
Phase 1: 3개 pane 병렬 → argument 수집 (hook)
Phase 2: 각 pane에 다른 프로바이더 argument 주입 (send-keys) → rebuttal 수집
Phase 3: judge pane에 전체 전달 → verdict 수집
```

### Relay
```
Pane A에 프롬프트 → hook 결과 수집
→ Pane B에 "A 결과 기반으로 보완해줘: {A결과}" 주입 → hook 결과 수집
→ Pane C에 "최종 정리해줘: {A결과} {B결과}" 주입 → 최종 결과
```

### Fastest
```
3개 pane 병렬 실행 → 첫 번째 hook done 감지 → 나머지 중단 → 결과 반환
```

## 추천 방향

**핵심 판단**: interactive.go의 ReadScreen 스크래핑을 hook 파일 시그널로 완전 교체.
pane은 제어(send-keys) + 모니터링(사람이 봄) 전용. 결과 수집은 hook이 담당.

### 실행 로드맵

**Phase 1 (즉시)**: Hook 파일 시그널 구현
- 3개 프로바이더용 hook/plugin 스크립트 작성 (shell script / TS plugin)
- 공통 파일 시그널 프로토콜 정의 (/tmp/autopus/{session-id}/)
- interactive.go에서 ReadScreen → 파일 감시로 전환
- cleanScreenOutput, ANSI 파싱 코드 제거

**Phase 2 (단기)**: Codex → opencode 전환
- orchestra 설정에서 codex → opencode 프로바이더 매핑
- opencode OAuth 인증 플로우 연동
- opencode plugin 자동 설정 구현

**Phase 3 (중기)**: 오케스트레이션 전략 고도화
- debate/relay에서 hook 결과를 다음 턴 프롬프트에 주입
- cmux pane 상태 표시 (프로바이더 라벨, 진행/완료)
- 결과 비교 뷰 (3개 프로바이더 응답 나란히)
