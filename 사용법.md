## 1. 설치하기

### 사전 준비: 프로젝트 폴더가 필요합니다

Autopus는 **프로젝트를 만들어주는 도구가 아닙니다.** 이미 있는 프로젝트 위에 AI 에이전트 환경(하네스)을 얹어주는 도구입니다. 따라서 **먼저 프로젝트 폴더가 있어야 합니다.**

#### 이미 프로젝트가 있는 경우 (바로 설치)

```bash
cd ~/my-existing-app    # 내 프로젝트 폴더로 이동
# 바로 아래 설치 명령어 실행
```

#### 새 프로젝트를 시작하는 경우 (폴더부터 만들기)

```bash
# 1. 폴더 생성
mkdir my-new-project
cd my-new-project

# 2. git 초기화 (필수)
git init

# 3. 프로젝트 초기화 (사용하는 언어에 맞게)
go mod init my-new-project    # Go 프로젝트
# 또는
npm init -y                   # Node.js 프로젝트
# 또는
python -m venv venv           # Python 프로젝트

# 4. 이제 Autopus 설치
```

설치 후 프로젝트 구조:

```
내 프로젝트 폴더/
├── src/                ← 내 코드 (이미 있어야 함)
├── package.json        ← 내 설정 파일
│
│  아래는 Autopus가 자동으로 추가하는 것들
├── .claude/            ← AI 에이전트 규칙/스킬/설정
├── autopus.yaml        ← Autopus 설정 파일
└── CLAUDE.md           ← AI가 프로젝트를 이해하는 문서
```

### macOS / Linux

터미널에서 프로젝트 폴더로 이동한 후 실행하세요:

```bash
cd 내프로젝트폴더                 # 반드시 프로젝트 폴더 안에서 실행
curl -sSfL https://raw.githubusercontent.com/Insajin/autopus-adk/main/install.sh | sh
```

### Windows

CMD 또는 PowerShell에서:

```powershell
cd 내프로젝트폴더                 # 반드시 프로젝트 폴더 안에서 실행
powershell -c "irm https://raw.githubusercontent.com/Insajin/autopus-adk/main/install.ps1 | iex"
```

### AI 코딩 도구 안에서 설치하기

Claude Code, Codex 등의 채팅창에 위 명령어를 그대로 붙여넣어도 됩니다. AI가 직접 실행해줍니다.

### 다른 설치 방법

```bash
# Homebrew (macOS)
brew install insajin/tap/autopus-adk

# Go가 설치되어 있다면
go install github.com/Insajin/autopus-adk/cmd/auto@latest

# 소스에서 직접 빌드
git clone https://github.com/Insajin/autopus-adk.git
cd autopus-adk && make build && make install

# 수동 설치 후에는 프로젝트 폴더에서 초기화 필요
cd 내프로젝트폴더 && auto init
```

### 설치 확인

```bash
auto version
```

버전 번호가 나오면 설치 완료입니다.

### 설치하면 무엇이 생기나요?

- `auto` CLI 바이너리가 설치됩니다
- 컴퓨터에 설치된 AI 코딩 도구를 자동으로 감지합니다
- 프로젝트에 필요한 설정 파일이 자동 생성됩니다

```
✓ Detected: claude-code, codex, gemini-cli
✓ Generated: .claude/rules/, .claude/skills/, CLAUDE.md
✓ Generated: .codex/, AGENTS.md
✓ Generated: .gemini/, GEMINI.md
✓ Created: autopus.yaml
```

## 2. 프로젝트 초기화

설치 시 자동으로 초기화되지만, 수동으로 할 수도 있습니다:

```bash
cd 내프로젝트폴더
auto init
```

이 명령어는:

- 설치된 AI 코딩 도구(Claude Code, Codex, Gemini CLI, OpenCode)를 감지합니다
- 각 플랫폼에 맞는 설정 파일을 생성합니다
- `autopus.yaml` 설정 파일을 만듭니다

## 3. 프로젝트 컨텍스트 설정

**이 단계가 가장 중요합니다.** AI는 매번 대화를 시작할 때 모든 기억을 잃습니다. 이 명령어로 AI가 프로젝트를 즉시 이해할 수 있는 "온보딩 문서"를 만들어줍니다.

```bash
/auto setup
```

이 명령어가 생성하는 파일들:

| 파일                            | 내용                                |
| ------------------------------- | ----------------------------------- |
| `ARCHITECTURE.md`               | 프로젝트 구조, 레이어, 의존성 맵    |
| `.autopus/project/product.md`   | 프로젝트가 뭘 하는지, 핵심 기능     |
| `.autopus/project/structure.md` | 디렉토리 구조, 패키지 역할          |
| `.autopus/project/tech.md`      | 기술 스택, 빌드 시스템, 테스트 전략 |
| `.autopus/project/scenarios.md` | 코드에서 추출한 E2E 시나리오        |

> 이 문서가 없으면 AI는 신입사원이 온보딩 없이 첫날 출근한 것과 같습니다.
> 있으면 매 세션마다 프로젝트를 완벽히 이해한 상태에서 시작합니다.

## 4. 기본 워크플로우 (3단계)

Autopus의 기본 사용법은 **Plan → Go → Sync** 3단계입니다.

### Step 1: Plan (기획)

원하는 기능을 자연어로 설명하면 됩니다:

```bash
/auto plan "로그인 페이지에 Google OAuth2 추가"
```

AI가 자동으로 생성하는 문서들:

```
.autopus/specs/SPEC-AUTH-001/
├── prd.md          # 제품 요구사항 문서
├── spec.md         # EARS 형식 요구사항
├── plan.md         # 작업 분해 + 에이전트 할당
├── acceptance.md   # 인수 테스트 기준 (Given/When/Then)
└── research.md     # 기술 조사 + 위험 분석
```

### Step 2: Go (구현)

생성된 SPEC ID를 넣으면 16개 에이전트가 자동으로 일합니다:

```bash
/auto go SPEC-AUTH-001 --auto --loop
```

파이프라인이 자동으로 실행됩니다:

```
Phase 1     Planner        기능을 작업 단위로 분해
Phase 1.5   Tester         실패하는 테스트 먼저 작성 (RED)
Phase 2     Executor x3    병렬로 코드 구현
Phase 2.5   Annotator      코드 문서화 태그 적용
Gate 2      Validator      빌드/테스트/린트 검증
Phase 3     Tester         커버리지 85% 이상 달성
Phase 4     Reviewer       코드 리뷰 + 보안 감사
```

결과:

```
╭────────────────────────────────────╮
│  Pipeline Complete!                │
│  SPEC-AUTH-001: Google OAuth2      │
│  Tasks: 5/5 │ Coverage: 91%       │
│  Review: APPROVE                  │
╰────────────────────────────────────╯
```

### Step 3: Sync (마무리)

문서를 업데이트하고 결정 이력을 커밋합니다:

```bash
/auto sync SPEC-AUTH-001
```

## 5. 한 줄로 끝내기 (빠른 경로)

3단계가 귀찮다면, 한 줄로 전부 할 수 있습니다:

```bash
/auto dev "로그인 페이지에 Google OAuth2 추가"
```

이 명령어는 `plan → go → sync`를 자동으로 연결합니다.
멀티모델 리뷰, 자가 치유, 팀 모드가 기본으로 활성화됩니다.

### 아이디어 탐색부터 하고 싶다면:

```bash
/auto idea "마이크로서비스로 전환해야 할까?" --multi
```

3개 AI 모델(Claude, Codex, Gemini)이 서로 토론하고,
아이디어를 ICE 점수(Impact, Confidence, Ease)로 평가합니다.

## 6. 자연어로 할 수 있는 것들

Autopus에서는 거의 모든 것을 **한국어 자연어**로 설명하면 됩니다.

### 기능 개발

```bash
# 새 기능 추가
/auto plan "회원가입 시 이메일 인증 기능 추가"
/auto plan "관리자 대시보드에 차트 추가"
/auto plan "결제 시스템에 카카오페이 연동"

# 한 줄로 바로 만들기
/auto dev "상품 검색에 필터링 기능 추가"
/auto dev "사용자 프로필 페이지 만들기"
```

### 버그 수정 (SPEC 없이 바로 가능)

```bash
/auto fix "로그인 페이지에서 500 에러 발생"
/auto fix "회원가입 버튼 클릭 시 아무 반응 없음"
/auto fix "모바일에서 메뉴가 안 열림"
```

AI가 자동으로:

1. 재현 테스트를 작성하고 (실패 확인)
2. 근본 원인을 분석하고
3. 최소한의 수정을 적용하고
4. 모든 테스트가 통과하는지 확인합니다

### 코드 리뷰

```bash
/auto review                    # 현재 변경사항 리뷰
/auto review --multi            # 3개 AI 모델이 교차 리뷰
/auto secure                    # 보안 취약점 스캔 (OWASP Top 10)
```

### 아이디어 탐색

```bash
/auto idea "웹소켓 대신 SSE를 써야 할까?" --multi
/auto idea "모노레포로 전환하면 좋을까?" --multi --ultrathink
```

### 코드 분석

```bash
/auto map                       # 프로젝트 구조 분석
/auto why "왜 이 패턴을 선택했지?"  # 과거 결정 이유 조회
/auto stale                     # 오래된 결정/패턴 감지
```

### 브라우저 자동화

```bash
/auto browse --url https://myapp.com/settings   # 웹 페이지 접근성 분석
/auto canary                                     # 배포 후 자동 건강 체크
/auto canary --url https://myapp.com             # 특정 URL 대상
```

### 자유 형식 (SPEC 없이 바로)

```bash
/auto 로그인 페이지에 2FA 추가해줘
```

Autopus가 자동으로 복잡도를 판단합니다:

```
Triage ────────────────────────────
  Request: "로그인 페이지에 2FA 추가해줘"
  Complexity: HIGH → /auto idea --multi (recommended)
```

## 7. 상황별 사용 예시

### "간단한 기능을 빠르게 추가하고 싶어"

```bash
/auto dev "GET /healthz 헬스체크 엔드포인트 추가"
```

### "큰 기능을 체계적으로 만들고 싶어"

```bash
# 1. 아이디어 단계 - AI 3개가 토론
/auto idea "OAuth2 Google + GitHub 로그인" --multi --ultrathink

# 2. 기획 단계 - SPEC 문서 생성
/auto plan "OAuth2 Google + GitHub 로그인"

# 3. 구현 단계 - 16개 에이전트가 병렬로 작업
/auto go SPEC-AUTH-001 --auto --loop --team

# 4. 마무리 - 문서 업데이트
/auto sync SPEC-AUTH-001
```

### "버그를 고치고 싶어"

```bash
/auto fix "주문 완료 후 이메일이 안 감"
```

### "코드 리뷰를 받고 싶어"

```bash
/auto review                    # 기본 리뷰
/auto review --multi            # AI 3개가 교차 리뷰
```

### "보안 점검을 하고 싶어"

```bash
/auto secure
```

### "중간에 파이프라인이 중단됐어"

```bash
/auto go SPEC-AUTH-001 --continue    # 마지막 체크포인트에서 재개
```

### "배포 후 확인하고 싶어"

```bash
/auto canary --url https://myapp.com
```

## 8. 주요 명령어 모음

### AI 코딩 도구 안에서 쓰는 명령어 (슬래시 커맨드)

| 명령어               | 설명                                      |
| -------------------- | ----------------------------------------- |
| `/auto plan "설명"`  | 기능 요청을 SPEC 문서로 변환              |
| `/auto go SPEC-ID`   | SPEC 기반 파이프라인 실행 (16개 에이전트) |
| `/auto dev "설명"`   | plan + go + sync 한번에 (추천)            |
| `/auto fix "설명"`   | 버그 수정 (SPEC 없이 바로)                |
| `/auto idea "설명"`  | 멀티모델 브레인스토밍                     |
| `/auto review`       | TRUST 5 코드 리뷰                         |
| `/auto secure`       | OWASP Top 10 보안 감사                    |
| `/auto sync SPEC-ID` | 문서 업데이트 + Lore 커밋                 |
| `/auto map`          | 프로젝트 구조 분석                        |
| `/auto setup`        | 프로젝트 컨텍스트 문서 생성               |
| `/auto why "질문"`   | 과거 결정 이유 조회                       |
| `/auto browse`       | 브라우저 자동화                           |
| `/auto canary`       | 배포 후 건강 체크                         |
| `/auto stale`        | 오래된 결정/패턴 감지                     |
| `/auto experiment`   | 자율 실험 루프 (성능 최적화 등)           |
| `/auto 자유형식`     | 자연어로 바로 요청 (자동 분류)            |

### 터미널에서 직접 쓰는 명령어 (CLI)

| 명령어                   | 설명                                 |
| ------------------------ | ------------------------------------ |
| `auto init`              | 프로젝트 초기화, 플랫폼 감지         |
| `auto update`            | 하네스 업데이트 (규칙/스킬/에이전트) |

| `auto doctor`            | 환경 건강 진단                       |
| `auto version`           | 현재 버전 확인                       |
| `auto connect`           | AI 프로바이더 연결 마법사            |
| `auto status`            | SPEC 대시보드 (진행 상황)            |
| `auto telemetry cost`    | 파이프라인 비용 추정                 |

| `auto arch`              | 아키텍처 분석                        |
| `auto lore query "질문"` | 결정 이력 검색                       |

### 플랫폼별 명령어 형식

| 플랫폼                   | 형식        | 예시                     |
| ------------------------ | ----------- | ------------------------ |

## 9. 옵션 플래그 설명

명령어 뒤에 붙여서 동작을 제어할 수 있습니다.

### 실행 모드

| 플래그       | 의미                            | 언제 쓰나                           |
| ------------ | ------------------------------- | ----------------------------------- |
| `--auto`     | 사람 승인 없이 자동 실행        | 파이프라인을 자율적으로 돌릴 때     |
| `--loop`     | 실패 시 자동 수정 후 재시도     | 게이트 통과까지 반복하고 싶을 때    |
| `--team`     | 팀 모드 (Lead/Builder/Guardian) | 복잡한 기능을 역할 분담으로 할 때   |
| `--solo`     | 서브에이전트 없이 혼자 실행     | 간단한 작업을 빠르게 할 때          |
| `--multi`    | 여러 AI 모델이 토론/합의        | 중요한 결정의 품질을 높이고 싶을 때 |
| `--continue` | 중단된 파이프라인 재개          | 크래시 후 이어서 하고 싶을 때       |

### 품질 모드

| 플래그               | 의미                                  | 비용 |
| -------------------- | ------------------------------------- | ---- |
| `--quality balanced` | 작업별 적절한 모델 자동 선택 (기본값) | $    |
| `--quality ultra`    | 모든 에이전트를 최고 모델로 실행      | $$$  |
| `--ultrathink`       | 깊은 사고 모드 활성화                 | $$   |

### 기획 옵션

| 플래그               | 의미                                    |
| -------------------- | --------------------------------------- |
| `--skip-prd`         | PRD 생략하고 바로 SPEC 작성 (작은 기능) |
| `--prd-mode minimal` | 간소화된 PRD 생성                       |
| `--from-idea`        | 브레인스토밍 결과에서 자동 연결         |

### 조합 예시

```bash
# 완전 자율 모드 (가장 많이 쓰는 조합)
/auto go SPEC-ID --auto --loop

# 최고 품질 팀 모드
/auto go SPEC-ID --auto --loop --team --quality ultra

# 멀티모델 리뷰 포함 전체 자동
/auto go SPEC-ID --auto --loop --multi

# 간단한 작업 빠르게
/auto go SPEC-ID --solo
```

## 10. 업데이트 방법

### CLI 바이너리 업데이트

```bash
auto update --self
```

GitHub에서 최신 버전을 다운로드하고, SHA256 체크섬을 검증한 후, 바이너리를 교체합니다.

### 프로젝트 하네스 업데이트

```bash
auto update
```

규칙, 스킬, 에이전트 파일을 최신 템플릿으로 재생성합니다.
사용자가 직접 수정한 내용은 `AUTOPUS:BEGIN`~`AUTOPUS:END` 마커 밖이면 보존됩니다.

### 한 번에 둘 다

```bash
auto update --self && auto update
```

## 11. 문제 해결

### "환경에 문제가 있는 것 같아요"

```bash
auto doctor
```

AI 도구 설치 여부, API 키 설정, 필수 도구 등을 자동 진단합니다.

### "AI 프로바이더 연결이 안 돼요"

```bash
auto connect
```

3단계 마법사가 안내합니다: 감지 → 설정 → 연결 확인

### "파이프라인이 중간에 멈췄어요"

```bash
/auto go SPEC-ID --continue
```

YAML 기반 체크포인트에서 마지막 완료 단계부터 재개합니다.

### "뭘 해야 할지 모르겠어요"

```bash
/auto 하고 싶은 것을 자유롭게 적으세요
```

Autopus가 자동으로 복잡도를 판단하고 적절한 명령어를 추천합니다.

## 12. 용어 정리

| 용어                      | 의미                                                               |
| ------------------------- | ------------------------------------------------------------------ |
| **하네스 (Harness)**      | AI 에이전트들이 일하는 환경/틀. 규칙, 게이트, 파이프라인을 제공    |
| **SPEC**                  | 기능 요구사항 문서. `SPEC-AUTH-001` 같은 ID가 붙음                 |
| **EARS**                  | 요구사항 작성 형식 (Easy Approach to Requirements Syntax)          |
| **파이프라인**            | Plan → Test → Build → Validate → Review 자동 실행 흐름             |
| **에이전트**              | 특정 역할을 맡은 AI (Planner, Executor, Tester 등 16개)            |
| **워크트리 (Worktree)**   | git의 독립 작업 공간. 여러 에이전트가 동시에 코드를 수정할 때 사용 |
| **RALF Loop**             | RED→GREEN→REFACTOR→LOOP. 품질 게이트 실패 시 자동 수정 반복        |
| **TRUST 5**               | 코드 리뷰 기준: Tested, Readable, Unified, Secured, Trackable      |
| **Lore**                  | 결정 이력. 왜 이 선택을 했는지 커밋에 기록                         |
| **오케스트레이션**        | 여러 AI 모델을 동시에 실행하여 토론/합의시키는 것                  |
| **게이트 (Gate)**         | 다음 단계로 넘어가기 전 통과해야 하는 품질 검증                    |
| **서킷 브레이커**         | 무한 반복을 방지하는 안전장치 (최대 재시도 횟수 제한)              |
| **AX (Agent Experience)** | AI 에이전트가 코드를 잘 이해할 수 있도록 설계하는 개념             |
| `--multi`                 | Claude, Codex, Gemini 3개 모델이 동시에 작업                       |
| `--auto --loop`           | 사람 개입 없이 자동 실행 + 실패 시 자가 치유                       |
| `--team`                  | 에이전트를 Lead/Builder/Guardian 역할로 나눠서 팀으로 운영         |

## 예시: 실전 프로젝트에서 사용하기

> **명령어를 치는 곳이 두 군데입니다. 헷갈리지 마세요!**
>
> | 구분 | 어디서 치나 | 예시 |

> | **터미널 명령어** | macOS 터미널, Windows CMD/PowerShell | `npm install`, `git init`, `curl ...` |
> | **AI 채팅 명령어** (`/auto`) | Claude Code, Codex 등 AI 도구의 **채팅창** | `/auto dev "..."`, `/auto fix "..."` |

### 프론트엔드: React

/auto setup

# ---- AI 채팅창에서 실행 ----
